# Nitnem Authentic — техническая документация и концепт

## Назначение

Nitnem Authentic — Android-first приложение для чтения Нитнема в редакционной рамке первых 13 ангов Sri Guru Granth Sahib. Текущий проект уже является рабочим Jetpack Compose-прототипом: он загружает локальный JSON-пакет, показывает произведения и анги, даёт переключатели текстовых слоёв и открывает информационные материалы из assets.

Документ описывает текущее состояние, будущий технический контур и TODO. Формат намеренно близок к рабочему product/technical concept: сначала пользовательские сценарии, затем архитектура, данные, разработка и backlog.

## Пользовательские истории

- Как пользователь я хочу открыть приложение и сразу видеть текст Нитнема, чтобы начать чтение без регистрации, сети и настройки.
- Как пользователь я хочу листать анги вперёд и назад, чтобы читать первые 13 ангов последовательно.
- Как пользователь я хочу открыть меню произведений, чтобы перейти к Джап, Со Дар, Сохиле или другому разделу без ручного поиска анга.
- Как пользователь я хочу включать и выключать транслитерацию, основной перевод, художественный слой, контекст и обоснование перевода, чтобы читать в подходящем мне режиме.
- Как пользователь я хочу видеть строки Рахао визуально выделенными, чтобы понимать смысловой центр шабда.
- Как пользователь я хочу читать информационные разделы о концепции Ek Granth Ek Panth Ek Maryada и символе приложения, чтобы понимать редакционную рамку.
- Как пользователь я хочу иметь словарь и пояснения терминов, чтобы не терять смысл при встрече с сикхскими и гурмат-терминами.
- Как пользователь я хочу пользоваться приложением офлайн, чтобы текст был доступен в дороге, в гурдваре или без стабильного интернета.
- Как пользователь я хочу получать обновления контента, чтобы исправления перевода и новые редакционные материалы появлялись без полной переустановки приложения.
- Как редактор я хочу хранить контент в версионированном JSON-пакете, чтобы можно было проверять, пересобирать и обновлять приложение предсказуемо.
- Как разработчик я хочу иметь стабильную схему контента, чтобы Android, будущий iOS-клиент и возможный API читали один и тот же источник.

## Текущее состояние

### Android-приложение

Основной код расположен в `nitnem_mobile/app/src/main/java/org/ksd/nitnem/MainActivity.kt`.

Сейчас реализовано:

- Jetpack Compose UI в одном файле `MainActivity.kt`.
- Material 3 светлая тема и палитра чтения, близкая к DOCX-цветам перевода.
- `ModalNavigationDrawer` с разделами:
  - произведения из JSON;
  - информационные блоки из content pack;
  - локальные markdown/assets-страницы: обновления, Ek Granth, символ приложения, словарь;
  - список ангов.
- Reader screen на `LazyColumn`.
- Верхняя панель с выбранным ангом/произведением, кнопками предыдущий/следующий и меню.
- Переключатели видимости слоёв:
  - транслитерация;
  - основной перевод;
  - художественный слой;
  - контекст;
  - обоснование перевода;
  - автор.
- Карточки строк с Gurmukhi, roman display, переводом, художественным слоем, контекстом и комментариями.
- Разворачивание/сворачивание строки.
- Выделение Рахао через отдельный цвет рамки и текста.
- Минимальный markdown-renderer для локальных статей.
- Загрузка изображений из assets.
- Загрузка JSON-пакета из `app/src/main/assets/nitnem_ru_ksd_v1.json`.

### Контент

Контентная подготовка живёт в `nitnem_app/`.

Ключевые файлы:

- `nitnem_app/export_nitnem_content.py` — локальный экспорт контента.
- `nitnem_app/content/nitnem_ru_ksd_v1.json` — generated content target.
- `nitnem_app/content_schema_v1.md` — схема контент-пакета.
- `nitnem_app/architecture.md` — начальное архитектурное направление.
- `nitnem_app/mvp_backlog.md` — исходный backlog.
- `nitnem_mobile/app/src/main/assets/nitnem_ru_ksd_v1.json` — пакет, который реально читает Android-приложение.
- `nitnem_mobile/app/src/main/assets/*.md` — локальные информационные страницы.
- `nitnem_mobile/app/src/main/assets/images/*` — изображения для информационных страниц.

### Сборка

Проект открывается как Gradle-проект из папки `nitnem_mobile`.

Ожидаемая среда:

- JDK 17.
- Android SDK API 36.
- Android Gradle Plugin 9.1.0.
- Gradle 9.3.1 или версия, которую ставит Android Studio.

Проверочный сценарий:

```bash
cd nitnem_mobile
./gradlew assembleDebug
```

Debug APK собирается в:

```text
nitnem_mobile/app/build/outputs/apk/debug/app-debug.apk
```

## Архитектурный концепт

### Целевое разделение

Текущий прототип намеренно простой. Следующий устойчивый шаг — разделить систему на слои:

```text
nitnem_mobile/
  app/
    ui/
      reader/
      drawer/
      info/
      settings/
      theme/
    data/
      content/
      assets/
    domain/
      models/
      reader/
      settings/

nitnem_app/
  exporters/
  schemas/
  editorial/
  content/
```

Долгосрочно можно вынести модели и парсинг в Kotlin Multiplatform:

```text
shared/
  models/
  content/
  reader/
  settings/
  sync/
androidApp/
iosApp/
```

### Границы ответственности

- `nitnem_app` — редакционная и генераторная зона. Здесь создаётся JSON, проверяется структура, готовятся инфоблоки и версии контента.
- `nitnem_mobile` — клиент чтения. Он не должен знать, как был создан перевод; он должен только читать стабильный content pack.
- `ksd_ai_translator.py`, `ksd_build_knowledge.py`, `ksd_knowledge.db` и родственные скрипты — зона KSD-аналитики и генерации знания. Их не нужно смешивать с runtime-логикой мобильного приложения.
- Будущий API/WordPress-плагин должен читать уже подготовленные данные, а не запускать переводчик во время пользовательского запроса.

### Модель данных

Основной контракт описан в `nitnem_app/content_schema_v1.md`.

Смысловой минимум:

- `schema_version` — версия схемы JSON.
- `content_version` — версия конкретного пакета.
- `package_id` — идентификатор набора.
- `source` — источник и диапазон ангов.
- `translators` — метаданные переводчиков.
- `banis` — пользовательские разделы/произведения.
- `angs` — анги, шабды, строки.
- `translations` — словарь переводов по translator id.

Android сейчас читает:

- `banis[0].title`;
- `banis[0].subtitle`;
- `banis[0].info_blocks`;
- `works`;
- `angs[].shabads[].lines[]`;
- `translations.ksd_ru.main`;
- `translations.ksd_ru.artistic`;
- `translations.ksd_ru.context_note`;
- `translations.ksd_ru.confidence_reason`;
- локальные markdown assets.

### Контентные обновления

MVP:

- JSON и markdown-файлы поставляются внутри APK.
- Приложение полностью работает офлайн.

Следующий этап:

- remote manifest с `package_id`, `schema_version`, `content_version`, checksum и URL пакета;
- скачивание нового JSON-пакета;
- проверка checksum/подписи;
- сохранение в локальное хранилище;
- fallback на bundled assets при ошибке.

### Будущий WordPress/API-плагин

Если нужен сайтовый плагин поверх тех же данных, он должен решать отдельную задачу: публикация и интерактивный просмотр content pack в браузере.

Целевой контракт:

```text
wp-json/ksd-nitnem/v1/manifest
wp-json/ksd-nitnem/v1/package/{package_id}
wp-json/ksd-nitnem/v1/ang/{number}
wp-json/ksd-nitnem/v1/work/{work_id}
```

Минимальные функции плагина:

- отдавать JSON без обращения к Python-генераторам;
- кэшировать ответы;
- показывать reader shortcode/block;
- переключать слои текста;
- давать ссылки на конкретный анг, шабд или строку;
- иметь режим обновления content pack администратором.

## Необходимые знания Python для проекта

Python нужен не в Android-runtime, а в генерации, проверке и подготовке контента.

Минимальный набор:

- базовый синтаксис Python 3: функции, классы, dataclass, списки, словари, исключения;
- работа с файлами через `pathlib.Path`;
- чтение/запись JSON через `json.load`, `json.dump`, `ensure_ascii=False`;
- работа с UTF-8, Unicode, Gurmukhi и русским текстом;
- CLI-скрипты через `argparse`;
- типизация: `list[str]`, `dict[str, Any]`, `Optional`, dataclasses;
- SQLite через `sqlite3` для локальной базы знания;
- DOCX-генерация через `python-docx`;
- тесты через `pytest`;
- аккуратная обработка частичных результатов: progress-файлы, backup JSON, повторный запуск без потери данных;
- умение отделять pure-функции трансформации данных от сетевых/API/LLM-вызовов.

Для этого проекта особенно важно:

- не ломать Gurmukhi и диакритику при нормализации;
- не править generated JSON вручную без понимания источника;
- добавлять валидаторы перед массовой генерацией;
- хранить схему данных явно и версионировать её;
- делать операции пересборки идемпотентными.
